Finch Architecture

Local-First AI Desktop Assistant — Architecture Overview

Finch uses an Electron dual-process architecture. The main process handles Agent orchestration and system operations; the renderer process provides the user interface. Communication happens over IPC, with all cross-boundary data types declared in a shared layer.

Renderer ⬇ IPC ⬆ Main Process ⬇ ⬆ Agent Core (pi) ⬇ ⬆ Electron Runtime

Process Layers

Renderer

Desktop UI built with React + TypeScript + Zustand. Components are organized by domain, using the desktopApi bridge to call backend capabilities without direct Node API access. Global state is split into domain-specific stores.

Main Process

The service container. All business logic — Agent orchestration, file I/O, session persistence, model management — is organized into independent service modules. The IPC handler layer stays thin, acting only as a router.

Agent Core

Runs in ~/.finch/pi/, managing session lifecycle, tool dispatch, and model communication. Sessions are persisted in JSONL format, organized by working directory.

Key Subsystems

Agent Runner

Accepts user input, calls LLMs for responses, orchestrates tool execution. Supports streaming output and background tasks. Each run has an independent context with no cross-contamination between final states.

Streaming Tool Orchestration Background Tasks

Memory System

Layered memory architecture: short-term memory injected into session context, medium-term memory summarizes project patterns, long-term memory archives daily logs. An auto-distillation mechanism periodically compresses and evicts low-value content.

Layered Memory Auto Distillation Capacity Control

Skill System

Encapsulates AI capabilities into registrable, triggerable skill modules. Each Skill defines trigger rules, execution context, and permission declarations. Built-in skills include document processing, PDF editing, PPT creation, front-end design, and surveys.

Plugin Architecture Trigger Matching Extensible

Permission System

Three-level permission model: Ask (confirm each action), Accept Edits (auto-approve), Developer (full access). File operations and trusted commands are auto-approved; high-risk operations require double confirmation. Supports batch reject.

Granular Control Sandbox Path Filtering

Model Layer

Unified model abstraction layer compatible with the OpenAI API protocol. Supports Anthropic Claude, OpenAI GPT, Ollama local models, and any third-party service that implements the OpenAI protocol. Model configuration is persisted locally.

Multi-Backend Unified Interface Local/Cloud

Session Management

Sessions are organized by working directory and persisted in JSONL format. Supports multi-window concurrent viewing of the same session, with real-time status broadcasting across windows. Sessions are resumable and searchable.

Multi-Window Real-Time Sync Persistent

Runtime & Data

All runtime data is stored in ~/.finch/, including Agent core data, session records, model configuration, and workspace state. No external service dependency — fully local-first.

Cross-platform support for macOS (ARM64 / Intel) and Windows (64-bit / 32-bit), leveraging Electron's cross-platform capabilities for build-once, deploy-everywhere.
